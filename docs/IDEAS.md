# IPCYolo-rk3588 想法记录

## 2026-04-08

### 关键阶段出现 fatal error 时应统一停机

当前情况：
- `RequestPipelineStop()` 会统一设置 `runtime.stop_requested = true`，并 `StopAll()` 所有队列。
- 目前只有信号打断和极少数初始化失败场景会调用 `RequestPipelineStop()`。
- `DecodeStage` 发生致命错误时，当前更多是 `decode_queue().stop()` 后直接 `return`，不会广播到上下游。
- `InferenceStage` 单帧推理失败时只是打日志后 `continue`，没有区分“单帧失败”和“工作线程已经不可恢复”的情况。

问题：
- 如果解码线程提前退出，上游 `CaptureStage` 可能仍在继续读取输入。
- 对于长流，例如 RTSP，如果上游不自然停下，主线程在 `join()` 时可能会等待很久。
- 目前“只关本阶段队列”的停机方式，不一定能保证上下游都及时收敛退出。

建议：
- 明确区分“可忽略的单帧失败”和“需要终止整条流水线的 fatal error”。
- 对于后者，在阶段内部直接调用 `RequestPipelineStop(runtime_, queues_)`，让所有阶段收到统一停机信号。
- 为每个阶段列出一份错误策略表，约定哪些错误是 `continue`，哪些是 `break + RequestPipelineStop`。

预期收益：
- 任何关键阶段不可恢复失败时，上下游都能更快地一起收敛退出。
- 避免“下游已经停了，上游还在狂跑”的状态，减少排查卡退出问题的成本。
- 为后续做更清晰的线程退出协调和错误上报机制打基础。

## 2026-03-27

### 模型类型的内部表示

当前情况：
- `--model-type` 在命令行里以字符串形式传入。
- `AppOptions::model_type` 当前存储为 `std::string`。
- 内部逻辑通过 `"yolov8"` 这类字符串比较来分支。

建议：
- 命令行输入层继续保留字符串形式，便于使用。
- 在解析阶段把字符串转换为 `enum class ModelType { kYoloV5, kYoloV8 };`。
- 将 `AppOptions::model_type` 改为存储 `ModelType`，而不是 `std::string`。
- 增加 `ParseModelType()`、`ToString(ModelType)` 这类辅助函数。

预期收益：
- 避免非法模型类型进入运行时逻辑。
- 去掉 pipeline 和 inference 中重复的魔法字符串判断。
- 后续新增模型类型时，更容易借助编译期检查发现遗漏分支。

后续可继续推进：
- 用基于枚举的分支替换重复的 `UseYolov8Model()` 字符串判断。
- 将模型类型相关的辅助逻辑集中收敛到 `app_options` 中。

### 类别数不应硬编码为 `OBJ_CLASS_NUM`

当前情况：
- `postprocess.h` 中将 `OBJ_CLASS_NUM` 固定为 `80`。
- YOLOv5 后处理通过 `PROP_BOX_SIZE = 5 + OBJ_CLASS_NUM` 参与输出解码。
- 标签文件加载也按 `OBJ_CLASS_NUM` 固定读取和存储。
- YOLOv8 后处理已经能从输出 tensor 动态推导类别数，但标签表仍带有固定类别数假设。

问题：
- 自定义模型的类别数不一定是 80。
- 如果标签文件行数少于或多于 `OBJ_CLASS_NUM`，当前实现无法正确完整表达。
- 对 YOLOv5 来说，这不只会影响标签显示，还可能导致输出解码维度假设错误。

建议：
- 将“模型类别数”从宏常量改为运行时配置，或从模型输出动态推导。
- 将标签数量与模型类别数解耦，不再强制按固定 80 类加载。
- 对 YOLOv5 / YOLOv8 分别梳理后处理所需的类别数来源，并统一收口到上下文或配置结构中。

预期收益：
- 支持自定义类别数模型，而不局限于 COCO 80 类。
- 避免标签文件数量与模型类别数不一致时出现越界、截断或错误显示。
- 为后续支持更多模型变体打下更稳妥的接口基础。

### 支持软件解码回退

当前情况：
- 现有解码链路固定为 `FFmpeg demux + MPP 硬解码`。
- `DecodeStage` 只持有 `MppDecoder`。
- 如果 `mpp_init` 或后续 MPP 解码流程失败，当前实现会直接初始化失败或结束流程。
- 仓库中没有接入基于 FFmpeg `avcodec_send_packet / avcodec_receive_frame` 的软件解码路径。

问题：
- 当硬件解码不支持某种码流、设备环境异常，或目标平台不具备 MPP 能力时，程序无法退化运行。
- 这会让项目对平台和码流格式的依赖过强，降低可移植性和容错能力。

建议：
- 抽象统一的解码器接口，让 `DecodeStage` 不直接绑定 `MppDecoder`。
- 增加一个基于 FFmpeg 的软件解码实现，作为硬解失败后的可选回退路径。
- 支持通过配置显式指定 `mpp`、`ffmpeg-sw` 或 `auto` 解码模式。

预期收益：
- 在硬解失败时仍能继续运行，提升鲁棒性。
- 更方便在非 RK 平台上调试整条流水线。
- 为后续做跨平台或混合解码策略预留接口空间。

### flush 冲洗时剩余帧的渲染

当前情况：
- 当前流水线已经支持在 EOS 到来后继续 drain 解码器和推理阶段剩余结果。
- 但 flush 阶段剩余帧的渲染和统计收尾策略还没有被单独梳理成明确约束。

建议：
- 明确 flush 阶段剩余帧是否必须全部渲染，还是允许在某些模式下跳过显示只保留统计。
- 梳理 EOS 后各阶段的关闭顺序，确保“最后几帧”不会因为队列提前关闭而丢失。
- 为 flush 过程补一组时序说明或状态图，方便后续继续优化退出流程。

预期收益：
- 让 EOS / drain / render 的收尾行为更可预测。
- 减少“最后几帧是否应该显示出来”这类边界问题。
- 为后续优化退出延迟和尾帧完整性提供依据。
